## 1. What Is Claude Workflows

A **portable, spec-driven development workflow system** for Claude Code AI agents, distributed as a **Claude Code plugin**. Structured, repeatable processes for building features, fixing bugs, refactoring, testing, and shipping releases.

**Core Principles**: Spec before code. Brainstorm before plan. Plan before implement. State persists across sessions. Quality gate before every PR. Language-agnostic (build/test commands auto-detected). Configurable git flow.

**What You Get**: 23 workflow skills (as plugin slash commands), 5 brainstorming techniques, orchestration rules enforced on every execution, review checklists (general + language-specific), coding rules per language, multi-session persistence, and a single YAML config file.

## 2. Installation

**Prerequisites**: Git repository, Claude Code, `gh` CLI (for PR workflows).

### Step 1: Install the Plugin

```bash
claude plugin install ragaa07/claude-workflows
```

During installation, Claude Code prompts for **userConfig** values:

| Prompt | Default | Options |
|--------|---------|---------|
| `project_type` | `generic` | `android`, `react`, `python`, `swift`, `go`, `generic` |
| `team` | `none` | `android`, `ios`, `frontend`, `backend`, `none` |
| `git_main_branch` | `main` | Any branch name |
| `git_dev_branch` | `develop` | Any branch name |
| `commit_format` | `conventional` | `conventional`, `angular`, `simple` |

These values seed your project configuration and can be changed later with `claude plugin configure claude-workflows`.

### Step 2: Initialize the Project

```bash
# Inside Claude Code, run:
/claude-workflows:setup
```

The setup skill:

| Action | Result |
|--------|--------|
| Detects project stack | Scans for `build.gradle`, `package.json`, `go.mod`, etc. |
| Creates state directories | `.workflows/`, `.workflows/specs/`, `.workflows/history/`, `.workflows/learned/` |
| Generates config | `.workflows/config.yml` (from plugin defaults + your userConfig) |
| Updates `.gitignore` | Adds workflow state entries |

### Verify & Commit

```bash
ls .workflows/
cat .workflows/config.yml
git add .workflows/config.yml .gitignore
git commit -m "chore: initialize claude-workflows v3.0.0"
```

Then open Claude Code and run `/claude-workflows:start`.

## 3. Configuration

All configuration lives in `.workflows/config.yml`, generated by the setup skill from the plugin's `config/defaults.yml`.

**Four-tier resolution** (highest priority first):
1. **CLI flags** — per-invocation overrides (e.g., `--skip-brainstorm`)
2. `.workflows/config.yml` (project-level overrides)
3. Plugin userConfig (values set during `claude plugin install`)
4. Plugin defaults (`<plugin-root>/config/defaults.yml`)

```yaml
version: "3.0"

project:
  name: "My Project"
  type: "generic"                    # android | react | python | swift | go | generic
  language: "kotlin"                 # Drives rule/checklist selection
  team: ""                           # Team name (android, ios, frontend, backend)

git:
  branches:
    main: "main"
    development: "develop"
    feature: "feature/{name}"
    bugfix: "bugfix/{name}"
    hotfix: "hotfix/{name}"
    release: "release/v{version}"
  commits:
    format: "conventional"           # conventional | angular | simple
    types: [feat, fix, refactor, test, docs, chore, style, perf]
    scopes: true
  pr:
    base_branch: "develop"
    draft: false
  merge:
    strategy: "squash"               # squash | merge | rebase
    delete_branch: true
  protected: ["main", "master", "develop"]

workflows:
  new-feature:
    require_spec: true
    require_tests: true
    require_brainstorm: true
    max_plan_phases: 10
    adaptive: true
  hotfix:
    base_branch: ""                  # Empty = use git.branches.main
    require_tests: true
    max_lines: 15
  release:
    changelog: true
    tag_format: "v{version}"
  review:
    checklist: "auto"                # auto | none | <path>
    size_thresholds: { small: 10, medium: 30, large: 50 }
  brainstorm:
    default_depth: "standard"        # quick | standard | deep

state:
  directory: ".workflows"
  history: true

skills:
  aliases:
    build: "new-feature"
    fix: "hotfix"
    ship: "release"

quality:
  self_review: true
  focused_gate: true
  proportional: true

telemetry:
  enabled: false                     # opt-in
  file: ".workflows/telemetry.jsonl"
```

**Tips**: Start with `project.name`, `project.type`, and `git.branches`. The `project.type` drives which coding rules and review checklists are loaded automatically.

## 4. The Workflow Lifecycle

Every workflow follows a state machine:

```
IDLE -> GATHER -> SPEC -> BRAINSTORM -> PLAN -> BRANCH -> IMPLEMENT -> TEST -> PR -> DONE
                                                             |
                                                        REPLAN -> back to IMPLEMENT
```

| Phase | What Happens | Output |
|-------|-------------|--------|
| **GATHER** | Collect requirements (Jira, Figma, user, spec file) | `.workflows/<feature>/01-gather.md` |
| **SPEC** | Generate formal specification | `.workflows/<feature>/02-spec.md` |
| **BRAINSTORM** | Explore approaches, evaluate trade-offs | `.workflows/<feature>/03-brainstorm.md` |
| **PLAN** | Phased implementation plan | `.workflows/<feature>/plan.md` |
| **BRANCH** | Create feature branch per git config | Git branch |
| **IMPLEMENT** | Write code phase-by-phase with build checks | Source code + commits |
| **TEST** | Write and run tests | Test files + results |
| **PR** | Push branch, create pull request | GitHub PR |
| **DONE** | Archive state, report summary | History entry |

**Orchestration rules** (`<plugin-root>/skills/_orchestration/RULES.md`) apply to every execution: phase output documents written to `.workflows/<feature>/`, state updated at every transition, build/test commands auto-detected, quality gate enforced, workflow chaining supported.

**Skippable phases**: Set `require_spec/brainstorm/tests: false` in config or pass `--skip-brainstorm`.

**Decision points**: After SPEC, BRAINSTORM, PLAN, UI implementation, and before PR. The workflow never proceeds without your approval.

**REPLAN**: Triggered when compilation fails 3+ times, plan is wrong, or you request a change. Stops, documents the issue, re-plans, and asks for approval.

## 5. Quality Gate

Two mechanisms enforce code standards across all workflows. Both are **bundled in the plugin** and loaded dynamically based on `project.type` and `project.language`.

### Coding Rules (`<plugin-root>/rules/`)

Language-specific DO/DON'T patterns loaded before implementation phases. Available for: Kotlin, TypeScript, Python, Go, Swift, React, Compose. Selected by `project.language` in config.

### Review Checklists (`<plugin-root>/reviews/`)

Before creating any PR, the agent self-checks against:
1. `general-checklist.md` (always loaded)
2. Language-specific checklist (auto-detected from `project.language`)
3. Team checklist (if defined)

Critical and High severity violations must be fixed before the PR is created.

```yaml
quality:
  self_review: true
  review_checklists: ["general-checklist.md"]
```

## 7. Workflows In Depth

### New Feature

The most comprehensive workflow. Takes a feature from idea to merged PR.

```
/claude-workflows:new-feature <name> [--from-ticket <url-or-id>] [--from-figma <url>] [--from-spec <path>] [--skip-brainstorm]
```

**GATHER**: Collects from Jira (MCP), Figma (MCP), spec file, or interactive questions. Multiple sources combinable. **SPEC**: Generates spec with user stories, acceptance criteria, scope, technical requirements, edge cases. **BRAINSTORM**: 2-4 approaches evaluated with structured techniques; produces decision document. **PLAN**: Phased implementation in `.workflows/<feature>/plan.md` with files, details, build checks, commit messages per phase. Also generates `tasks/todo.md`. **BRANCH**: Per git config. **IMPLEMENT**: Phase-by-phase with build checks, commits, and state updates. Coding rules loaded from `<plugin-root>/rules/`. REPLAN on repeated failures. **TEST**: Targets all new code, reports coverage. **PR**: Quality gate checklists run, then `gh pr create`.

Decision points after SPEC, BRAINSTORM, PLAN, UI implementation, and before PR.

### Extend Feature

```
/claude-workflows:extend-feature <feature-name> <extension-description> [--skip-brainstorm]
```

**Conservative** workflow: existing behavior must be preserved. Prefers adding new files over modifying existing ones. Uses SCAMPER brainstorming technique. VERIFY-COMPAT phase ensures all existing tests still pass. If existing tests break, the new code is fixed (not the tests).

### Refactor

```
/claude-workflows:refactor <target> [--scope files|module|feature] [--goal "description"]
```

Full dependency graph analysis, behavioral contracts documented before changes, incremental migration where **every step compiles and passes tests independently**. Brainstorm uses Constraint Mapping and Trade-off Matrix (blast radius as highest weight).

### Hotfix

```
/claude-workflows:hotfix <description> [--crashlytics <issue-id>] [--log <path>] [--branch <production-branch>]
```

Optimized for **speed**. No brainstorming, no spec. Absolute minimum change, branches from production branch. Diagnose → Fix → Regression-Test → PR → Cherry-Pick plan.

### Test

```
/claude-workflows:test <target> [--coverage <pct>] [--type unit|integration]
```

Targets: `class:Name`, `file:path`, `module:name`, `feature:name`. Analyzes public API, dependencies, branches, edge cases. Writes tests following project conventions. Default coverage: 90%.

### Review

```
/claude-workflows:review <pr-number>
```

Fetches PR via `gh`, categorizes changes by layer, checks architecture/quality/security/performance/coverage/standards. Generates inline comments with severity levels. You choose which to submit.

### Release

```
/claude-workflows:release <version>
```

Changelog from commits since last tag, version bump (language-agnostic detection), release branch, PR to production, tag command.

### CI Fix

```
/claude-workflows:ci-fix [--run <run-id>] [--pr <pr-number>]
```

Fetches failure details via `gh`, classifies (compile/test/lint/dependency/config/timeout), applies targeted fix, pushes, provides watch command.

### Migrate

```
/claude-workflows:migrate <type>
```

Types: `dependency`, `api-version`, `architecture`, `database`. Each step compiles and tests pass independently. Type-specific guidance for version comparison, endpoint mapping, pattern migration, schema migration.

### New Project

```
/claude-workflows:new-project [--path <directory>] [--preset <android|web|rust|go|python>]
```

Detects project type from build files, CI system, and git conventions. Generates `.workflows/config.yml` and directory structure.

## 8. Brainstorming System

Runs standalone (`/claude-workflows:brainstorm <topic>`) or as part of any workflow.

### 5 Techniques

| Technique | Method |
|-----------|--------|
| **Trade-off Matrix** | Score options against weighted criteria (complexity, maintainability, performance, testability, time) |
| **Six Thinking Hats** | 6 perspectives: Facts, Feelings, Risks, Benefits, Creativity, Process |
| **SCAMPER** | Substitute, Combine, Adapt, Modify, Put to other use, Eliminate, Rearrange |
| **Reverse Brainstorm** | "How could this fail?" — invert failures into mitigations |
| **Constraint Mapping** | Filter by hard/soft constraints before analysis |

### Depth Levels

| Depth | Options | Techniques | When |
|-------|---------|-----------|------|
| `quick` | 2 | Trade-off Matrix only | Simple decisions |
| `standard` | 3 | 1 technique + Trade-off Matrix | Most features |
| `deep` | 4+ | Multiple techniques + Trade-off Matrix | Architecture, high-risk |

**Defaults**: Technique selection is based on depth level — standalone brainstorm selects technique per depth config. extend-feature explicitly uses SCAMPER, refactor uses Constraint Mapping. All levels include Trade-off Matrix for final evaluation.

## 9. Git Flow Management

Handled by the `/claude-workflows:git-flow` skill, reading from `.workflows/config.yml`.

- **Branches**: Pattern-based (`feature/{name}`, `release/v{version}`)
- **Commits**: Conventional, Angular, or Simple format
- **PRs**: Template with `{summary}`, `{changes}`, `{test_plan}`, `{ticket}` variables
- **Protected branches**: Warns on direct commits, suggests feature branch
- **Hotfix cherry-pick**: Provides command to backport fix to development branch

## 10. Multi-Session State Management

Three-layer persistence solves context loss between sessions:

| Layer | File | Purpose |
|-------|------|---------|
| State | `.workflows/current-state.md` | Phase, progress, context notes |
| Spec + Plan | `.workflows/<feature>/`, `.workflows/<feature>/plan.md` | What and how (with `[x]` progress) |
| Git | Branch, log, diff | Committed and uncommitted code |

**Session recovery** (`/claude-workflows:resume`): Reads state, spec, plan; verifies branch; reports phase and progress; continues from where it left off.

**Context notes** in the state file capture decisions and discoveries throughout the workflow.

**One active workflow** at a time. Options: resume, pause (preserved as `paused-<name>.md`), or abandon (archived to `history/`).

## 11. Todo & Lessons System

**`tasks/todo.md`**: Operational checklist generated from the plan, updated during implementation.

**`tasks/lessons.md`**: Captures mistakes — what went wrong, correct pattern, prevention rule. Written on corrections, REPLAN triggers, failures. Read at session start and before each implementation phase.

| File | What | Updated |
|------|------|---------|
| `tasks/todo.md` | Day-to-day checklist | After each step |
| `.workflows/<feature>/plan*.md` | Architectural plan | Planning and REPLAN |
| `.workflows/current-state.md` | Session metadata | Phase boundaries |

## 12. Sub-Agent Strategy

Workflows can leverage sub-agents to parallelize work and protect the main context window. The following is a recommended strategy — Claude will apply sub-agents at its discretion based on task complexity.

| Phase | Sub-Agents | Purpose |
|-------|-----------|---------|
| GATHER | 1 Explore | Scan codebase |
| BRAINSTORM | 2 parallel | Analyze + find alternatives |
| PLAN | 1 Plan | Design implementation |
| IMPLEMENT | None | Main thread writes all code |
| TEST | 1 background | Run test suite |
| REVIEW | 1 expert | Analyze PR diff |

Rules: One task per agent. Main thread writes all code. Max 3 concurrent. Results summarized, not dumped raw.

## 14. Upgrading

```bash
claude plugin update claude-workflows
```

The plugin auto-updates skills, templates, rules, and checklists. Your project-level config (`.workflows/config.yml`), state files, and tasks are preserved.

To reconfigure plugin settings:

```bash
claude plugin configure claude-workflows
```

To check the current version:

```bash
cat $(claude plugin path claude-workflows)/VERSION
```

## 15. Troubleshooting

| Problem | Solution |
|---------|----------|
| Skills don't appear | Run `claude plugin list` and verify `claude-workflows` is installed |
| Commands not found | All commands are namespaced: `/claude-workflows:<skill-name>` |
| Git operations fail | Verify branch names in `.workflows/config.yml` |
| PR creation fails | Run `gh auth status` |
| State corrupted | Delete `.workflows/current-state.md`, restart |
| Quality gate not running | Check `quality.self_review: true` in `.workflows/config.yml` |
| Wrong rules loaded | Check `project.language` in `.workflows/config.yml` |
| Config missing | Run `/claude-workflows:setup` to regenerate |

### Reset

```bash
# Reset workflow state
rm .workflows/current-state.md .workflows/paused-*.md

# Reinstall the plugin entirely
claude plugin uninstall claude-workflows
claude plugin install ragaa07/claude-workflows

# Re-initialize the project
# Then run /claude-workflows:setup inside Claude Code
```

# View orchestration rules
cat $(claude plugin path claude-workflows)/skills/_orchestration/RULES.md
